Project Overview:

We would like to build a fault-tolerant, containerized postgresql as a service solution.

 

Assume you are given a number of VMs (e.g. RHEL 8 VMs) that can run Postgresql instances as containers. We'd like you to

 

1) write APIs that can:

- create new DBs on the existing PGSQL Instances: accept parameters for requesting a new pgsql DB and spin-up a new DB based on the provided parameters. The new DB will run on one of the postgresql instances running on the VMs you are provided, and a passive DB will need to be provisioned on one of the other VMs that will shadow the primary DB and take over in case it fails. The API will return the needed parameters to the requesting user for them to be able to use the provisioned DB.

- delete existing DBs

- change parameters of existing DBs: e.g. increase capacity, reset password, ...

- report metrics about the DB usage/utilization/health

- spin up a new postgresql instance(s) on new VM(s) if all existing instances are highly utilized

 

For example some of the initial APIs to develop will have functionality like the below:

- List Postgres instances

- Get Postgres instance info

- Create a Postgres Instance

- Change a Postgres Instance's Settings

- Delete a Postgres Instance

- List databases on an instance

- Get info of a database

- Create a database on an instance

- Change settings of a database

- Delete a database

 

2) write UIs that can:

- display usage of the service for sys admins, e.g. how many databases provisioned, space usage and transaction count of each DB, etc...

- showback usage of each DB to tenants

- UI tech stack selection is up to the team

 

3) Stretch goals:

- build a service that can span multiple clouds (e.g. a private OpenStack cloud and Google Cloud)

- build a service that can run PGSQL VM instances as well as PGSQL container instances
