Project Overview:

Background:  

FoundationDB [https://foundationdb.apple.com/], [https://www.foundationdb.org/files/fdb-paper.pdf ] is a distributed open source key value database with strong ACID properties. It is a core port of Apple and Snowflake’s production clouds, and used in many other places. It is a layered system, in that new functionality can be added on top of the core foundation, which can transform it into a database with different semantics. For instance the ”record layer” was introduced to give SQL-like operations to FoundationDB. The ”document layer” provides MongoDB semantics. There is also a block layer, which enables mounting a block device over the database.

Besides strong consistency, FoundationDB is famous for having outstanding  automated tests which employ a unique deterministic simulation engine. These tests keep the database in a bulletproof state with little regression as it evolves. Upper layers inherit FDB’s reliability in distributed systems and ACID properties, which is why the database makes a good “foundation”.

Project Specifics:

In this project, you will write a simple file system layer over FoundationDB. The end result will be a distributed file system with very good consistency properties. You will leverage FoundationDB’s tests, which should mean the file system will be very resilient to failures in a distributed setting.

To get the file system running, you will use FUSE. This is a framework that intercepts Linux kernel API calls and routes them through your user space code which will call the FoundationDB client API. The interface will be stateless.

Implementation steps

  1. Work with the mentor to design a mapping between files and directories to key value.
  1. See [kvfs] for an example discussion on mapping.
  2. Some operations such as links or renames are more difficult and can be deferred, but should be called out in the final report.
  1. Run FoundationDB locally or in the cloud, and write the client layer using the FoundationDB interface. The client layer will probably wish to leverage FoundationDB’s “directory later”. [https://apple.github.io/foundationdb/developer-guide.html ]
  2. Connect FUSE to the client layer
  3. Write tests. The overall system should enjoy the same reliability as FoundationDB.
  4. Do performance measurements. The distributed file system will be slower than a purpose-built one, but very reliable.
