Project Overview:

Background:

Continuous Integration (CI) is a technique for regularly exercising a code base with the goal of identifying bugs or regressions quickly by running a set of tests on a code base regularly. It is used by most modern software development teams and is part of a tool set that developers can use to set themselves up for success.

The Linux kernel is the dominant kernel powering the internet today and it is used for many research projects when access to a “real world” kernel is required.  The OS research literature has many papers that demonstrate the power of unikernels or library OS’s. However, options for using the well tested Linux kernel code base as a unikernel have been limited.

Unikernel Linux (UKL, https://github.com/unikernelLinux) is a set of out-of-tree patches on top of the Linux kernel and glibc which allows you to compile existing programs (with the emphasis on servers) combined with Linux and glibc into a single kernel binary which can then be booted within a VM (eg. qemu) or on baremetal.

The Unikernel Linux project is under active development, but we do not currently have a CI setup so catching bugs in new features or regressions in existing features can be difficult.  Therefore, we would like to introduce CI which will test both patches pushed to the repository and (if we use them in future) push requests, alerting us early if we break the code.  However, this project will not only benefit the UKL project, our goal is to make it generic enough to be used by any OS projects at BU which target similar systems.

You should expect to learn a little bit about a large collection of subjects from this projects, including: the Linux Kernel, how ELF binaries are built, interacting with virtual machines using QEMU, the Mass Open Cloud (MOC), and more.

Project Specifics:

Since github offers CI already (“Github Actions”), it seems we may as well use that.

The students will need to develop some script(s) to compile new releases of UKL + glibc + a test program and then ensure it runs under qemu and produces the expected output.  All of this triggered by a Github Action.

Tommy Unger and Ali Raza wrote an outline script already which might be adapted or used for reference: https://github.com/unikernelLinux/ukl/blob/ukl/Makefile (NB: If you do not have access to this URL, ask Ali).

Richard Jones has extensive experience with running qemu appliances, and getting the output from them (eg. libguestfs architecture, qemu-sanity-check, qemu supernested) so I can help here.

Looking for performance regressions seems to be more complex than simply ensuring that the project doesn’t break, so we’re open to suggestions about how to do this.

The free tier of Github CI is quite limited.  It may be that all the work compiling these large projects (like Linux) and running under qemu uses up the free tier allowance quickly.  In that case we would need to look at how to provide “runners” hosted at BU to do the work.

Testing the kernel on baremetal is a stretch goal.  It will certainly not be possible to do this using Github-hosted CI, but it might be possible with self-hosted runners.  This will probably involve provisioning OpenStack Ironic instances or using IPMI directly.
