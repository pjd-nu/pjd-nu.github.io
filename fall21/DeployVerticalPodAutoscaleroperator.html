Project Overview:

Background:  

In Kubernetes, while actual workloads run within containers, Pods are in fact the smallest deployable unit of computing. This means that all compute resources (CPU/Memory) for any workload are specified at the Pod level. In Kubernetes, both forms of compute resources, CPU and Memory, are specified in similar ways, and are done so in a static manner. That is to say, when managing workloads via Pod, a human operator will first specify what compute resources this Pod will require to operate. It is not uncommon however, for a workload to demand more resources at peak times, while remaining mostly dormant during off peak hours. As one might expect, this results in either waste of compute resources or reduced pod performance.

The solution here is to move away from manually specifying the compute resources for a pod that are static, and opt for a more automated scaling solution. The ideal solution would identify when a Pod requires additional resources, and adjust it’s requirements accordingly during the lifetime of the Pod. The Vertical Pod Autoscaler (VPA) was designed to specifically do this. The VPA will automatically scale the resources for a pod based on its usage trends. The goal of this project is to transform Operate First OCP clusters from manually specifying compute resource requirements for pods, and to instead use the VPA instead. Thus having all workloads automatically scale up/down their resources on a need basis.

Project Overview:

Background:  

With the adoption of Kubernetes (K8S), comes the need to manage a plethora of resources. With the rise of Operators and API extensions in K8S, there is no limit to the number of different types of resources one can have available in their K8S cluster. Many such resources are often privileged, and should only be touched by system admins. Luckily for this, K8S comes with builtin RBAC (Role based access control) support out of the box. This RBAC feature provides strong mechanisms to gate access at the resource level. For instance, with RBAC one could easily prevent a certain group of users from deploying Pods in a particular namespace.

But what if we want to go a bit more granular? What if we want to gate access based on the controls, settings, or configurations of a resource? For example, what if we want all Pods to have a specific label, or what if we want to limit the number of replicas for a specific group of users per deployment? Unfortunately, this level of granularity in gating resources is not possible with RBAC.

We can however use Open Policy Agent (OPA) Gatekeeper to accomplish. OPA is an open source service that can gate inputs against user defined policies. The “Gatekeeper” portion offers K8S integration, allowing for a powerful way to control access to a cluster. There are many ways in which Operate-First would like to leverage OPA. This project will focus on a subset of such cases.
